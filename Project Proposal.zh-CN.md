## 1 引言

### 1.1 编写目的
本计划旨在为 **Code-Repository** 项目的开发工作制定全面、有序的执行方案，明确各阶段任务、人员分工、进度安排与资源需求。本文档的预期读者包括：项目投资人、项目经理、前后端开发工程师、测试工程师、运维人员及相关审批管理人员。

### 1.2 背景
a. **待开发的软件系统的名称**：Code-Repository（在线代码与文件托管平台）  
b. **任务提出者**：Vito-Lonius 项目组  
   **开发者**：Vito-Lonius 及其开源社区贡献者  
   **用户**：个人开发者、中小型团队、需私有化部署代码仓库的企业  
   **实现平台**：支持部署在 Linux 云服务器、物理机或本地虚拟化环境，通过 Docker 容器化运行

## 2 项目概述

### 2.1 工作内容
本项目开发须完成以下主要工作：
1. **需求分析**：明确定义用户注册登录、仓库管理、文件操作、在线预览、代码质量检测、消息通知等详细需求。

3. **编码实现**：完成后端核心服务（用户、仓库、文件、通知）、前端管理界面、Docker 镜像打包。
4. **测试**：单元测试、集成测试、性能测试、安全测试（含 SonarQube 扫描）。
5. **文档编写**：用户手册、管理员手册、API 文档、部署文档。
6. **部署与运维支持**：提供 Docker Compose 一键部署方案及基本运维脚本。

### 3.5 关键问题
1. **Office 文档在线预览转换性能**：使用 LibreOffice 转换可能耗时较长，影响用户体验。影响：文件预览模块验收可能延迟。应对方案：启动异步转换队列，增加转换节点。
2. **SonarQube 集成复杂度**：需要深入 SonarQube Scanner API 并异步解析结果，技术难度较高。影响：代码质量检测功能的质量与稳定性。应对：提前进行技术预研，必要时修改简化方案。
3. **并发与存储瓶颈**：在大文件上传、流媒体预览时，单节点可能出现带宽/IO瓶颈。影响：系统性能指标无法达成。应对：采用 MinIO 分片上传、CDN 缓存策略，并做水平扩展预留。
4. **人员变动风险**：主力后端若中途离开，项目进度将严重受阻。应对：关键模块补充文档，培养其他成员交叉接手能力。

## 5 专题计划要点
1. **分合同计划**：本项目暂不涉及子合同分包，所有模块自行开发。若后期需引入前端UI外包，需另行制定外包管理计划。
2. **开发人员培训计划**：前端开发人员需要培训 MinIO 文件流处理相关知识；后端人员需熟悉 SonarQube API。计划在 2026 年 6 月第一周集中培训。
3. **测试计划**：包括单元测试（强制，覆盖率≥80%）、接口自动化测试、压力测试（并发 50 逐步增至 200）、安全渗透测试。详细计划在 D1 阶段末尾发布。
4. **安全保密计划**：用户密码使用 bcrypt 加密，API 强制 JWT 校验，敏感配置使用环境变量或 Secret 管理。定期进行 SonarQube 安全扫描。
5. **质量保证计划**：代码需经过 Code Review，强制 CI 门禁（编译、测试、SonarQube 检查），Release 分支保护。
6. **配置管理计划**：使用 Git 进行版本控制，仓库分支策略采用 main/develop，标签化发布。Docker 镜像版本与 Git Tag 同步。
7. **用户培训计划**：2026 年 12 月进行，面向管理员及核心用户，包含操作演示与常见问题解答，时长 2 小时。
8. **系统安装计划**：交付时提供详细的 Docker Compose 安装脚本，目标在 30 分钟内启动完整服务。安装手册将给出多种环境参数示例。
